## Part 1：RAG 三级检索管道——不只是「搜文档 + 喂 LLM」

### 从 IR 到 RAG：接口契约的建立

之前 Day 11 学的信息检索（倒排索引→TF-IDF→BM25→稠密向量）提供了语料库里的「搜东西」能力，但 RAG 要做的是在「检索」和「生成」之间定义一个稳定的接口契约。这个契约的核心就是：

```text
answer = LLM(prompt_template + retrieved_chunks + user_query)
```

retrieved_chunks 就是检索侧的输出格式 => 生成侧的输入格式，这是整个管道的接口边界。只要这个接口不动，检索侧换 BM25/向量/图都行，生成侧换 DeepSeek/GPT/Claude 也行。这种感觉非常接近后端里「接口定义与实现分离」的设计原则——定义好上下游的契约，中间件可以任意替换。

### 三级管道的分工逻辑

今天花最多时间在理解「为什么是三级，不是两级或一级」：

- **第一级（BM25 关键词）**：精确匹配专用。数字、API 名、错误码、专有名词——这些向量检索的软肋恰是 BM25 的强势范围。查 `NullPointerException` 不会有语义变体，就是要精确命中。类比 B+Tree 索引做等值查询。

- **第二级（稠密向量）**：语义近似专用。同义改写、跨语言匹配、概念联想。查「线程池调优」能找到「HikariCP pool sizing」，BM25 做不到。类比全文索引的模糊匹配。

- **第三级（Reranker）**：精排专用。Cross-Encoder 对 query-document pair 做完整前向计算，纠正前两级的排序错误。但不能做全量（计算太贵），只对 top-N（N≈50-100）做。类比数据库 ORDER BY 后在应用层用 Comparator 二次排序。

### RRF 融合算法的直觉

Reciprocal Rank Fusion 的核心公式：`RRF_score(d) = Σ 1/(k + rank_i(d))`，k=60。

这个算法的美感在于「不需要调权重」——不像加权求和那样纠结「BM25 权重 0.7 还是 0.3」。它尊重每个检索器的排名，排名靠前的天然获得更高分数。Elasticsearch 8.x 内置支持，生产上可以直接用。

### 文档切分的直觉修正

读完 Anthropic 的建议后一个关键认知转变：chunk_size 不是独立参数，它和 top_k 联动决定检索到的内容总量能否「刚好填好」上下文窗口。公式是：

```text
chunk_size × top_k ≤ 上下文可用空间 × 检索文档占比
```

另外学到了「短文档不切分」这个反直觉结论——当文档本身已经接近最优 chunk 大小时，不切分就是最好的切分。这个认知对后续 Day 14 的 chunking 实验有直接指导意义。

## Part 2：知识图谱认知 + 结构化/非结构化混合查询

> **图解决的是「关系」，向量解决的是「相似」**

这是今天最核心的认知锚点：

| 问题类型 | 适合技术 | 示例 |
|----------|----------|------|
| 「这个概念是什么？」 | Vector RAG | HikariCP 怎么配置？ |
| 「A 和 B 有什么关系？」 | GraphRAG | HikariCP 依赖哪些 Spring Boot 自动配置条件？ |
| 「这件事的上下游是什么？」 | GraphRAG | 连接池参数调整会影响哪些组件？ |

向量检索本质上在做「语义空间里的最近邻查找」——适合回答「XX 是什么」「YY 怎么配」这类事实型问题。但当你问「HikariCP 依赖了哪些 Spring Boot 条件注解，这些注解之间有什么互斥关系」时，要的不是相似文本，而是一个关系链——这时候图遍历远比向量检索有效。

从 Java 工程师的视角，这非常像 SQL JOIN 和 Elasticsearch full-text search 的区别：JOIN 能回答关系型问题，ES 能回答内容型问题。没有人会用 JOIN 做全文搜索，也不会用 ES 做复杂的多表关联查询——同样的道理，GraphRAG 和 Vector RAG 是互补关系，不是替代关系。

### GraphRAG 构建流程（架构认知层面）

```text
构建侧：原始文档 → 实体抽取(LLM/spaCy) → 关系构建(三元组) → 知识图谱存储(Neo4j)
查询侧：用户Query → 实体识别 → 图遍历(1-hop/2-hop) → 子图上下文 → LLM生成
```

这块今天只做到「能画出架构图，能讲清楚每阶段做什么」的程度，没有要求自己动手实现。GraphRAG 的工程复杂度主要在实体对齐和关系消歧上——同一个实体在不同文档里可能有多种表述，需要做实体链接（Entity Linking），这在生产上是远比「搭个 Neo4j 写 Cypher」更花精力的活。

### 结构化 + 非结构化混合查询

这一步是把前两 Part 的思路落到混合查询场景里：用户问一个既有结构化数据需求、又有非结构化文档需求的问题，怎么做。

```text
用户Query: "上个月销售额最高的5个产品？"
  ├── Text-to-SQL → MySQL → 结构化数据（金额、排名）
  └── 向量检索   → ChromaDB → 非结构化文档（产品描述、用户评价）
                        ↓
               结果融合（LLM 拼接回答）
```

核心难点不在 Text-to-SQL 或向量检索本身——这两块各自有成熟方案——而在「融合」这一步。LLM 需要被告知哪些信息来自数据库（权威数值）、哪些来自文档（可解释性描述），否则容易产生交叉污染——把文档里的一个约数当成精确数据来引用。这是很多 RAG demo 被诟病「说得头头是道但数字是编的」的根因。

## Part 3：上下文工程——把 Context Window 当 JVM 堆来管

这是今天最「爽」的一部分——因为从 Java 工程师的视角，上下文管理和 JVM 堆管理有大量同构的直觉。

> **核心认知：上下文不是「无限大袋子」**

以前我对 Context Window 的直觉是「128K 够大，往里塞就行」。今天被纠正：窗口大小 ≠ 可用质量。塞得越满，信息密度越低，模型的注意力被稀释得越厉害——这跟 JVM 堆越大 GC 时间越长是一个道理。

### 四块分配的动态策略

上下文窗口需要根据任务类型调整四块的比例：

| 场景 | System | 历史消息 | 检索文档 | 工具输出 |
|:-----|:------:|:------:|:------:|:------:|
| 单轮问答 | 10% | 0% | 80% | 10% |
| 多轮对话 | 10% | 50% | 20% | 20% |
| Agent 编排 | 15% | 20% | 10% | 55% |
| 长文档分析 | 5% | 5% | 85% | 5% |

这个分配表的核心逻辑是：**上下文里的「主角」由任务类型决定**。单轮问答时文档是主角，Agent 编排时工具输出是主角。没有一成不变的公式，但需要有「主动设计分配比例」的意识，而不是被动地往里塞。

### 渐进式披露：按需注入，不是一次性注册

这个原则在后端系统里太熟悉了——就是懒加载。不要让 Agent 一次性看到 50 个工具的定义，而是按当前执行阶段分层注入：意图理解阶段只传语义解析工具，查询阶段只传检索工具，格式化阶段只传导出工具。

对应的知识分层也很有意思：热知识（每次都用的）放 System Prompt，温知识（偶尔参考）放向量库按需检索，冷知识（极少用）放规则引擎触发时才加载。类比：热知识 = static final 常量，温知识 = Redis 缓存，冷知识 = 数据库按需查询。

### 上下文重置：GC for Context

这是上下文工程里最 Java 的一个概念映射：

| JVM 概念 | 上下文工程对应 |
|----------|-------------|
| 堆最大容量 (-Xmx) | 模型窗口上限 (128K token) |
| 年轻代 (Eden) | 最新一轮消息 |
| 老年代 (Old Gen) | System Prompt + 持久摘要 |
| Minor GC | 滑动窗口截断旧消息 |
| Full GC | 上下文重置（全局压缩） |
| 内存泄漏 | 无截断的累积对话历史 |

三种重置策略里，「滑动摘要」最像 Kafka 的日志压缩——每 N 轮对话生成一个摘要，用摘要替代原始历史，保留语义但不保留冗余。这是生产上多轮对话系统的标配，单靠「记住最近 N 轮」远远不够。

### 文件系统分流：大数据落盘

当检索结果超过 2000 token，不要硬塞进 prompt——写到文件系统，prompt 里只传路径引用。LLM 需要详细信息时，通过文件读取工具来拿。这和 JVM 里「小对象放栈上、大对象放堆上、只传引用」是一个思路。

## 今日关键认知总结

> **RAG 的本质不是「搜 + 生成」，而是「检索和生成的接口契约」**。接口定义好，检索侧和生成侧各自演进，互不耦合。
2. **三级管道每一级解决一个特定问题**：BM25 做精确匹配、向量做语义近似、Reranker 做精排。任何一级都不是银弹，组合才能覆盖完整查询谱系。
3. **知识图谱不替代向量 RAG，而是互补**。图解决关系型问题（多跳推理），向量解决相似性问题（语义匹配），生产上两者共存。
4. **上下文是有限资源，需要主动管理**。四块动态分配 + 渐进式披露 + GC 策略 + 文件分流——这些不是锦上添花，是多轮对话系统能跑在 128K 窗口里的前提。
5. **Java 工程师做上下文管理有先天优势**——JVM 堆管理、GC 策略、懒加载、接口抽象这些概念在上下文工程里几乎是一一对应的。今天的学习过程更像是「把已有的工程直觉迁移到新领域」，而不是从零建立认知。
