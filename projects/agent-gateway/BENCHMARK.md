## Token 成本分析

| 指标 | 值 |
|------|-----|
| 总调用次数 | 12 |
| 总 Token | 5,878 |
| 平均 Token/次 | 490 |
| 工具调用 | 0 次（"你好" 无需工具） |
| 平均评估分 | 4.98/5 |

**单次调用成本估算**（DeepSeek V4 Pro 定价）：

| 项目 | 估算 |
|------|------|
| 输入 Token（System Prompt + 历史 + 用户消息）| ~290 token |
| 输出 Token（LLM 回复 + 评估 JSON）| ~200 token |
| 输入成本 | 290 × ¥1/1M = ¥0.00029 |
| 输出成本 | 200 × ¥2/1M = ¥0.0004 |
| **单次成本** | **≈ ¥0.0007（0.07 分）** |
| 月成本（10 万次） | ≈ ¥70 |
| 月成本（100 万次） | ≈ ¥700 |

**工具调用场景成本加成**：需工具调用时额外 +1 次 LLM 往返，成本约翻倍（~¥0.0014/次）。

**安全护栏开销**：正则 + Unicode NFKC 在 Python 层完成，不计入 LLM Token，额外延迟可忽略。

## 并发瓶颈分析

| 层次 | 瓶颈 | 表现 | 优化方向 |
|------|------|------|----------|
| 框架层 | 无 | P99 17ms，QPS 156 | — |
| LLM API | **主要瓶颈** | 2.6s/次，吞吐完全受限于 API 耗时 | 单次 token 数不可控，可做请求合并 |
| MCP 通信 | 无影响 | 本次未触发工具调用 | 工具调用场景下增加 1 次往返 (~2.6s) |
| 安全护栏 | 可忽略 | 纯内存正则，<1ms | — |
| Python GIL | 轻微 | 单 worker，CPU 密集时排队 | 多 worker 启动 (`--workers 4`) |

**核心结论**：Agent 网关的性能瓶颈 95% 在 LLM API 延迟，不在框架或代码。优化的重点不是代码调优，而是：
1. LLM API 并发控制（连接池 + 限流）
2. Token 缓存（相同 System Prompt 的 prefix caching）
3. 流式输出 → 用户体感延迟远低于实际耗时

## 和 Java 传统后端的对比认知

| | 传统 API 网关 | Agent 网关 |
|------|-------------|-----------|
| P99 延迟 | <50ms | 2,600ms |
| 主要耗时 | 网络 IO | LLM 推理 |
| 瓶颈 | 数据库连接池 | LLM API 并发 |
| 优化手段 | 缓存/索引/降级 | 流式/缓存/模型选型 |
| QPS 天花板 | 数千~数万 | 受限于 LLM API 配额 |

Agent 网关的工程挑战不是「快」，而是「可靠」——2.6s 的延迟里有一半时间在黑盒等 LLM 回复，重试、降级、超时、熔断这些传统微服务的稳定性手段在这里同样适用。
