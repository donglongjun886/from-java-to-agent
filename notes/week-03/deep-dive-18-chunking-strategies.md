# Chunking策略深度 — 从固定切分到语义感知的文档分块演进

> Chunking的黄金法则是"让每个chunk成为可独立理解的最小语义单元"——大小不是目标，自包含才是。

## 关键对比 / 架构认知

四种主流策略从简单到智能形成一条光谱。**固定大小切分**是最朴素的做法，按token数或字符数等距切分，通常配一个overlap做边界缝合。优点是零计算开销，缺点是毫不关心内容边界——一个Python函数的定义和函数体可能被拦腰截断在两块chunk里，检索时永远拼不回完整语义。**递归字符切分**是固定大小的变种：预设一个分隔符优先级（如 `\n\n` → `\n` → `。` → 空格），先按最高优先级分隔符切，单段太长就降级用下一级分隔符。这把简单性和语义边界做了折中——代码文档按空行切，普通文本按句号切，大多数场景够用且无需额外计算。

**语义切分**走的是另一个极端：计算相邻句子的embedding相似度，在相似度骤降处下刀。这完美捕捉了"话题切换"的边界，但问题在于：需要对文档中每个句子做embedding推理，相邻句子embedding的余弦相似度判断断点位置，耗时和开销是递归切分的数倍；而且相似度阈值需要调参，不同语料的最佳阈值不一致，工程上不稳定。**结构化切分**则是"先验知识驱动"——对Markdown按`#`标题层级切，对代码按AST的function/class节点切，对表格按`<tr>`行切。结构化切分利用了文档格式本身隐含的边界信息，对技术文档、API文档效果极好，但需要针对每种格式写解析器。

关于chunk size的Goldilocks区间：512-1024 token是目前的主流共识。太小（<256）时一个完整观点被分散到多个chunk，检索召回率低；太大（>2048）时chunk像"大杂烩"，即使用户问题只命中其中一句话也要把整块喂给LLM，既浪费上下文窗口又引入噪声干扰。这个选型逻辑和JVM内存分页如出一辙——4KB默认页是综合page fault频率和内存碎片找到的平衡点。

## Java 映射 + 面试话术

**Java 映射**：JVM的内存分页类比是最精准的映射。操作系统页面太大则一次缺页代价高、内存浪费（内部碎片），太小则频繁缺页（外部碎片），跟chunk size的trade-off完全同构。再类比一个：数据库索引设计。你把WHERE条件理解为用户查询，把一行记录理解为一个chunk。如果表设计太宽（chunk太大），一次查索引带回太多无关列；如果拆分过细（chunk太小），需要JOIN多表才能拼出完整信息。好的chunking就是好的数据库模式设计——找到"一个自包含的业务实体"的粒度。

**面试话术**："Chunking不是越大越好也不是越小越好，关键看每个chunk是不是一个自包含的语义单元。我们实际做得最多的是递归切分，因为它性价比最高——用分隔符优先级逐级回退，能兼顾简单性和语义边界。对于技术文档我偏好结构化切分，按Markdown标题或代码函数边界切，这样每个chunk天然就是一个知识单元。大小上512-1024 token是比较成熟的范围。这其实和JVM的分页机制逻辑很像——页面大小是缺页频率和内存利用率之间的平衡点，Chunk size是召回率和精确率之间的平衡点。"
