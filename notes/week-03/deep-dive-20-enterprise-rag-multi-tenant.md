# 企业级RAG多租户架构：索引隔离 vs 元数据过滤 vs Partition Key

> 多租户方案选型本质上是安全隔离度与运维复杂度的工程权衡——没有银弹，只有匹配业务规模的甜点区间。

## 关键对比 / 架构认知

企业级RAG上线前必须回答一个问题：多个租户的数据如何存储和检索？业界有三种主流方案，各有利弊。

**索引级隔离（Physical Isolation）**：每个租户在向量数据库中拥有独立的Collection或Index。这是安全边界最硬的方案——租户A的检索请求永远不会扫到租户B的数据，即使代码有bug也不会泄露。代价也很明显：租户数过百后，Collection数量爆炸，元数据管理、索引重建、备份恢复的操作成本按O(n)增长。适合金融、医疗等强合规场景，且租户数可控（<100）。

**元数据过滤（Logical Isolation）**：所有租户的数据共存于同一个Collection中，检索时通过 `WHERE tenant_id='X'` 过滤。运维极其简单，一个索引打天下。但风险在于：如果过滤条件因为代码缺陷未生效，检索结果会混入其他租户的数据——这是数据泄露事故。此外，单一Collection的索引质量会随数据异构性增加而下降。适用场景：SaaS产品早期，租户数据量小且安全等级不高。

**Milvus Partition Key（Physical + Logical）**：Milvus 2.2+引入的Partition Key机制，在逻辑上统一Collection，物理上按Partition Key自动分片存储。检索时指定Partition Key，Milvus只扫描对应分区，性能和安全的兼顾。这是目前生产环境的最优解：运维体验接近元数据过滤（一个Collection管理所有租户），安全隔离接近索引隔离（分区内数据物理隔离）。局限是依赖Milvus生态，换其他向量库需要重新设计。

| 维度 | 索引隔离 | 元数据过滤 | Partition Key |
|------|---------|-----------|---------------|
| 安全隔离度 | 高 | 低（依赖代码正确性） | 中高（多个租户共享物理分区，但通过分区裁剪实现逻辑隔离） |
| 运维复杂度 | 高（N个Collection） | 低 | 中 |
| 检索性能 | 高（索引小） | 中（索引大+过滤开销） | 高（分区裁剪） |
| 租户上限 | <100 | 无限制 | 1000+ |
| 资源利用 | 低（碎片化） | 高 | 高 |

逻辑租户数无上限，但物理分区数受限（<1024），多租户复用物理分区。

决策逻辑直截了当：租户数少但每家数据量大（如私有化部署）→索引隔离；SaaS多租户小数据快速验证→元数据过滤；生产环境追求稳妥→Partition Key。大多数企业在第一阶段用元数据过滤跑通MVP，第二阶段迁移到Partition Key。注意：Partition Key必须在创建Collection时指定，后期无法动态添加。从元数据过滤迁移到Partition Key需全量数据重建+双写迁移。

## Java 映射 + 面试话术

**Java类比**：索引隔离就是数据库分库——每个租户一个独立数据库实例，安全最硬但运维噩梦。元数据过滤就是单库加 `tenant_id` 行级过滤——简单但有SQL写漏的风险。Partition Key就是分表加自动路由——MyBatis拦截器根据租户ID自动选分区，物理隔离但逻辑统一。

**面试话术**：「我在设计企业级RAG系统时，多租户架构是第一个要解决的工程问题。从后端分库分表的经验出发，我直接排除了纯元数据过滤方案——代码里的过滤条件漏一次就是数据泄露。我们最终选了Milvus的Partition Key方案，本质上是数据库分区的思路：一个Collection逻辑统一，物理上按tenant_id自动分片，检索时Partition Key裁剪到单分区扫描。Partition Key的优势在于引擎层强制裁剪（降低人为漏写风险），应用层二次校验是纵深防御（Defense in Depth），所有方案都应具备。」
