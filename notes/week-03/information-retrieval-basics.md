# 信息检索基础

> 从 Java 后端视角理解检索系统：管好索引和排序，剩下的都是工程优化。
> 前置关联：[Prompt Engineering](./prompt-engineering.md)（检索结果最终要喂给 LLM）

---

## 1. 检索的本质：召回 + 排序

| 阶段 | 职责 | Java 类比 |
|------|------|-----------|
| **召回（Recall）** | 从海量文档中快速筛选候选集（百万 → 百级） | 数据库全表扫 → 走索引，过滤 99% 无关行 |
| **排序（Rerank）** | 对候选集精排（百级 → 十条），把最相关的排前面 | `ORDER BY relevance DESC LIMIT k`，但不是简单字段排序，而是模型打分 |

一个检索 pipeline 就是这两阶段串起来，无论后面多花哨（重排序、混合检索），骨架不变。

## 2. 稀疏检索三件套：倒排索引 → TF-IDF → BM25

这三个概念是递进关系，不是并列的三个方案。

### 2.1 倒排索引（Inverted Index）

把「文档→词」的映射反过来，建成「词→文档列表」的索引。

```
文档1: "Java 分布式事务"     →   倒排: "Java" → [doc1, doc3]
文档2: "Python 机器学习"              "分布式" → [doc1]
文档3: "Java 高并发"                  "事务" → [doc1]
```

**Java 类比**：相当于你给 MySQL 表的 `keyword` 列建了一个 B+Tree 索引，查 `WHERE keyword = 'Java'` 时不用扫全表。只不过倒排索引更激进——每个 term 直接指向包含它的文档 ID 列表（posting list），term dictionary 定位是 O(1) 的哈希查找，实际查询耗时取决于 posting list 长度和合并策略。

### 2.2 TF-IDF（词频-逆文档频率）

TF-IDF 给每个 (查询词, 文档) 对打一个**相关性分数**。一个词在一篇文档中出现越多次（TF），且在全局越少见（IDF），那它对该文档就越重要。

```
TF-IDF(词t, 文档d) = TF(t, d) × log(N / DF(t))

TF(t, d)  = 词t在文档d中出现次数 / 文档d总词数  （归一化，避免长文档占优）
IDF(t)    = log(总文档数 / 包含词t的文档数)      （常见词压低权重）
```

**Java 类比**：TF 像热点方法的调用次数，IDF 像这个方法在整个代码库中的稀有程度。`main()` 到处都有（IDF 低），但 `distributedLock()` 只在关键路径出现（IDF 高）→ 搜后者时权重更高。

### 2.3 BM25（Best Match 25）— TF-IDF 的工业加强版

BM25 是目前最主流的稀疏检索算法，ES 的默认相似度。它在 TF-IDF 基础上解决两个问题：

| 问题 | TF-IDF 的缺陷 | BM25 的修复 |
|------|-------------|------------|
| **TF 饱和** | 词频线性增长，一篇文档提 100 次「Java」得分就是提 10 次的 10 倍——不合理 | 对 TF 做饱和处理（`TF/(k1 + TF)`），词频超过一定阈值后边际收益递减 |
| **文档长度归一化** | 简单的词频/总词数，对长短文档不公平 | 引入 `b` 参数控制长度惩罚力度：短文档自动提权，长文档自动降权 |

**Java 类比**：TF 饱和 = 缓存命中率曲线，访问量从 0→1000 收益巨大，从 10000→11000 几乎没区别。BM25 的 `k1` 参数（通常 1.2~2.0）控制饱和速度，类似 Guava Cache 的 `maximumSize` 决定缓存何时开始驱逐。

**三个算法对比**：

| 维度 | 倒排索引 | TF-IDF | BM25 |
|------|---------|--------|------|
| 本质 | 数据结构（索引） | 相关性评分公式 | TF-IDF 的改进版 |
| 处理 TF 饱和 | 不支持 | 不支持 | 支持（饱和函数） |
| 处理文档长度 | 不处理 | 简单归一化 | 带参数控制 |
| 生产使用 | ES/Lucene 底层 | 早期搜索引擎 | ES 默认、Elasticsearch 标配 |

> 注：倒排索引是数据结构，非评分算法——上表对比的是基于它的评分能力。更准确的分层是：数据结构层（倒排索引 vs 稠密向量索引）和评分公式层（TF-IDF vs BM25）。此处为便于理解将三层放在一起对比。

## 3. 稠密检索：Embedding + 余弦相似度

稀疏检索的致命问题是**词不匹配**：搜「分布式锁」找不到写「Redisson」的文档——虽然语义完全相关，但没有共同词。

稠密检索的思路：把文本映射到高维向量空间，语义相近的文本向量距离也近。

```
文本 → Embedding Model（如 text-embedding-3-small）→ 向量 [0.023, -0.451, ..., 0.187]  (通常 768~1536 维)
查询向量 vs 文档向量 → 余弦相似度 → 排序
```

**余弦相似度**只关心向量方向的夹角，不关心长度：
```
cos(θ) = (A · B) / (|A| × |B|)

接近 1 = 语义几乎相同
接近 0 = 无关
接近 -1 = 语义相反（检索场景几乎不出现）
```

**Java 类比**：数据库的 `LIKE '%关键词%'` 只能做字面匹配，Embedding = 给每篇文档打个「语义标签」，不管用词如何，标签相近的就能召回。类似你在 ES 里做 synonym 映射，但 Embedding 不用人工维护同义词表。

**生产落地**：存向量用 ChromaDB / Milvus / pgvector，检索用 ANN（近似最近邻）算法如 HNSW——不做精确搜索，百万向量 10ms 内返回 top-100。

## 4. 混合检索：BM25 + 向量 + RRF

### 4.1 为什么两者互补

| 场景 | BM25 | 向量检索 |
|------|------|---------|
| 精确关键词匹配（序列号、错误码、API 名） | 强——倒排索引直查 | 弱——语义向量可能发散 |
| 模糊语义查询（同义词、口语化表达） | 弱——词不匹配直接零分 | 强——向量空间兜底 |
| 处理新领域术语（训练集外的词） | 强——倒排不依赖预训练 | 弱——Embedding 模型没见过就抓瞎 |
| 长文档检索 | 一般——词频稀释 | 强——整段语义压缩为一个向量 |
| 计算开销 | 低（CPU 即可） | 高（GPU 推理 + 向量索引） |

**一句话**：BM25 保证「搜什么返回什么」（精确），向量保证「搜意思返回意思」（泛化）。两者互补，缺哪个都可能翻车。

### 4.2 RRF（Reciprocal Rank Fusion）融合算法

把两路排序结果融合成一列，不依赖原始分数的绝对值（BM25 和余弦相似度的分值不可比）：

```
RRF_score(文档d) = Σ 1/(k + rank_i(d))

k ≈ 60（平滑参数，行业经验值。k 越小，高排名文档在融合时权重越大；k 越大，各排名贡献越趋于均匀。k=60 是折中值，同时防止 rank=0 导致除零。）
```

单路排名第 1 = 贡献 1/(60+1) ≈ 0.0164，排名第 2 = 1/(60+2) ≈ 0.0161——排名越高贡献越大，但不是线性衰减。

**流程**：两路各取 top-100 → RRF 融合 → 取 top-20 → 可选重排序模型 → top-k 返回。

**Java 类比**：类似两个微服务各返回一个排序列表，你用加权合并（不是简单 union）得到最终结果。RRF 就是那个合并策略，不关心各自的评分量纲。

## 5. 与 RAG 的关系

```
RAG 质量 = 检索质量 × 生成质量

检索决定上限（喂进去什么），生成决定下限（别胡说八道）
```

| 环节 | 出问题时的影响 |
|------|---------------|
| 检索漏召（Recall 低） | LLM 看不到正确答案，生成必然错误 |
| 检索误召（Precision 低） | 喂一堆噪声，LLM 被干扰，作答走偏 |
| 排序不准 | 正确答案在 top-20 但不在 top-3，LLM 上下文窗口被浪费 |

RAG 不是一个「增强搜索」的技术，是一个「检索结果增强生成」的技术。检索是前置条件，不是可有可无的部件。

## 6. 面试要点

**能画出来的架构图**：

```
用户查询 → [查询改写/扩展] → BM25 召回（倒排索引 + ES）
                            ↘     → RRF 融合 → 重排序（Cross-Encoder）→ Top-K → LLM
                              向量召回（Embedding + ChromaDB/Milvus） ↗
```

**能讲清楚的核心论点**：

1. 「混合检索比纯向量好」——因为 BM25 对精确词匹配有不可替代的优势，向量在模糊语义上有优势，两者覆盖不同类型的查询场景，RRF 融合互取所长
2. 「BM25 不是过时算法」——它是工业界二十年磨合出来的最优稀疏检索方案，饱和函数和长度归一化解决了 TF-IDF 的核心缺陷
3. 「检索是 RAG 的天花板」——检索漏了，LLM 再强也答不对；检索噪声多了，LLM 会被带偏。先把检索做好再谈 Prompt

**需要记住的关键数字（面试加分）**：

| 参数 | 推荐值 | 记法 |
|------|--------|------|
| BM25 k1 | 1.2 ~ 1.5 | 控制 TF 饱和速度 |
| BM25 b | 0.75 | 控制文档长度惩罚 |
| RRF k | 60 | 平滑参数 |
| Embedding 维度 | 768 / 1024 / 1536 | 取决于模型 |
| ANN top-k | 通常取 100 | 候选集规模 |
| Chunk 大小 | 256~512 tokens | 检索粒度 |
