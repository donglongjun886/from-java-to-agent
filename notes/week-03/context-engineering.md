# 上下文工程（Context Engineering）

> 学习笔记（2026-06-28）。核心认知：**Prompt 决定单轮质量，Context 决定多轮天花板。上下文是 Agent 的「工作内存」，组织方式比总量重要。**

---

## 1. 什么是上下文工程

上下文工程不是写 prompt，而是**设计「在什么时机、把什么信息、以什么形式」放进模型视野**的策略。

```
┌─────────────────────────────────────┐
│  上下文窗口（Context Window）         │
│  ┌───────────────────────────────┐  │
│  │ System Prompt   ~5%           │  │
│  │ 历史消息        ~30%          │  │
│  │ 检索文档        ~40%          │  │
│  │ 工具输出        ~25%          │  │
│  └───────────────────────────────┘  │
│  窗口有硬上限（如 128K token）        │
└─────────────────────────────────────┘
```

> 误区：把上下文当「无限大袋子」往里塞。现实：窗口满了就丢信息，丢哪块你控制不了。

| Java 类比 | 上下文工程概念 |
|-----------|--------------|
| JVM 堆内存 | 上下文窗口（有上限，需管理） |
| 线程栈 | 单次对话的调用链信息 |
| ThreadLocal | 会话级状态（用户偏好、临时变量） |
| 方法参数 | 每轮注入的工具输出 / 检索结果 |
| GC Root | System Prompt（始终存活，不被回收） |

---

## 2. 渐进式披露（Progressive Disclosure）

**核心原则：不要一次性塞所有信息，分层按需加载。**

```
第一层（必现）：  System Prompt — 角色 + 核心约束（始终在窗口）
第二层（按需）：  工具定义 — 只传当前步骤可能用到的工具
第三层（懒加载）：参考文档 — 检索命中后才注入，不命中不占位
第四层（临时）：  工具执行结果 — 本轮用完即可丢弃
```

### 2.1 工具定义瘦身

```python
# ❌ 一刀切：50 个工具全部注册 → token 爆炸
tools = [search_tool, db_tool, email_tool, pdf_tool, ...]  # 50 个

# ✅ 渐进式：按当前 step 语义分组
# Step 1: 理解意图 → 只传语义解析类工具
# Step 2: 执行查询 → 只传检索/数据库类工具
# Step 3: 格式化输出 → 只传渲染/导出类工具
```

### 2.2 冷知识 vs 热知识

| 类型 | 存放位置 | 何时注入 |
|------|---------|---------|
| 热知识（每次必用） | System Prompt | 始终在线 |
| 温知识（偶尔参考） | 文档库 / 向量库 | 查询时检索注入 |
| 冷知识（极少使用） | 知识图谱 / 规则引擎 | 触发条件后才加载 |

类比：热知识 = 静态常量（全程在线），温知识 = 缓存（Redis），冷知识 = 数据库（按需查询）。

## 3. 上下文重置（Context Reset）

长对话多轮累积 → token 逼近上限 → 模型「遗忘」早期信息 → 回答质量退化。需要主动干预。

### 3.1 三种重置策略

```
策略 A：硬截断（Truncation）
  保留 System Prompt + 最近 N 轮 → 暴力删早期消息
  优点：简单  │  缺点：丢上下文，可能断档

策略 B：滑动摘要（Sliding Summary）  ← 推荐
  每 10 轮生成一次摘要，用摘要替代原始历史
  类似 Kafka 日志压缩 → 保留 key 的最新 value

策略 C：检查点重启（Checkpoint Restart）
  在关键节点保存上下文快照 → 下次从此快照恢复
  类比：游戏存档 / JVM 快照
```

### 3.2 什么时候该重置

| 信号 | 含义 | 操作 |
|------|------|------|
| `usage.total_tokens > 窗口 * 0.8` | 快满了 | 触发摘要压缩 |
| 用户说「从头来」 | 显式重置 | 清空历史，保留 System Prompt |
| 话题完全切换 | 隐式重置 | 保留摘要，切新上下文 |
| 回复质量明显下降 | 注意力稀释 | 缩短上下文，提高密度 |

## 4. 上下文窗口的四块分配

**核心公式**：

```
可用空间 = 模型窗口上限 - 输出预留
         = 128K - 8K = 120K (以 deepseek-chat 为例，实际以模型 max_tokens 设置为准)

四块分配（无银弹，按场景调比例）：
┌──────────────┬──────────┬──────────┬──────────┐
│   场景        │ System   │ 历史消息  │ 检索文档  │ 工具输出  │
├──────────────┼──────────┼──────────┼──────────┼──────────┤
│ 单轮问答      │   10%    │   0%     │   80%    │   10%    │
│ 多轮对话      │   10%    │   50%    │   20%    │   20%    │
│ Agent 编排    │   15%    │   20%    │   10%    │   55%    │
│ 长文档分析    │    5%    │   5%     │   85%    │    5%    │
└──────────────┴──────────┴──────────┴──────────┴──────────┘
```

### 4.1 各块的压缩技巧

```
System Prompt:  精炼到 5-10 句，只放「不变」的东西
历史消息:        超过 N 轮的旧消息 → 摘要替代
检索文档:        按相关度排序，低分文档不纳入
工具输出:        裁剪掉冗余字段，只保留 LLM 关心的部分
```

```python
# 工具输出瘦身：不要把 JSON 原封不动塞给 LLM
raw = tool_result  # {"status": "ok", "code": 200, "trace_id": "xxx", "data": {...}}

# ✅ 只保留有效信息
slim = {
    "rows": raw["data"]["total"],
    "items": raw["data"]["items"][:10]  # 裁剪分页
}
```

## 5. 分片策略与上下文质量

**Chunk 大小不是独立参数，它直接决定检索到的一批 chunk 能否「刚好填好」上下文窗口。**

### 5.1 三个联动参数

```
chunk_size  ×  top_k  ≤  上下文可用空间 × 检索文档占比

例如：
  500 token/chunk × 8 个  =  4000 token  → 适合 16K 窗口
  1000 token/chunk × 20 个 = 20000 token  → 需要 64K+ 窗口
```

### 5.2 大 chunk vs 小 chunk 的取舍

| 维度 | 大 chunk (1000+) | 小 chunk (200-500) |
|------|-----------------|-------------------|
| 语义完整性 | 高（不截断思路） | 低（可能断句） |
| 检索精度 | 低（噪声多） | 高（定位准） |
| 窗口占用 | 大 | 小 |
| 适合场景 | 长文档分析、法规 | 事实问答、FAQ |

> **本项目实验**：6 篇技术文档各约 140 token，top_k=3。chunk_size=128 → 11 nodes, Q1 Faithfulness 0.50（过度碎片化）；chunk_size=256 → 6 nodes（未实际切分）；不切分 → Q1 Faithfulness 0.67（最优）。结论：短文档保证语义完整性优先。分片没有普适最优解，需用实验数据验证。

### 5.3 实战：父子分片（Parent-Child Chunking）

```
检索用小 chunk (200 token) → 提高命中精度
返回用大 chunk (1000 token) → 保证语义完整

类比：数据库的覆盖索引 → 用索引定位，回表取完整行
```

## 6. 文件系统作为状态

**当上下文超过阈值，把中间结果写到文件，Prompt 里只放文件路径引用。**

### 6.1 模式

```python
# ❌ 错误：100 条搜索结果的全文塞进下一轮 prompt → 12K token
results = search(query)  # 100 条，每条 2000 字
next_prompt = f"分析以下结果：\n{json.dumps(results)}"  # 20K token

# ✅ 正确：大结果落盘，prompt 只传路径
with open("/tmp/search_result.json", "w") as f:
    json.dump(results, f)

next_prompt = """
结果已存储到 /tmp/search_result.json（共 100 条记录）。
请先分析文件结构，然后回答用户问题。
"""
# Prompt 只占 ~100 token
```

### 6.2 哪些该落盘

| 数据类型 | 阈值 | 处理方式 |
|---------|------|---------|
| 单条信息 | < 500 token | 直接放在 prompt |
| 中间计算结果 | 500-2000 token | 摘要放 prompt，原文落盘 |
| 大批量检索结果 | > 2000 token | 仅路径引用，文件工具读取 |
| 结构化数据 | 任意大小 | 落盘后用 SQL/Pandas 操作 |

> 类比：小数据放方法局部变量，大数据放堆上，只传引用。

## 7. Java 类比：上下文管理 ≈ JVM 堆内存管理

```
┌────────────── JVM 堆 ──────────────┐    ┌────── 上下文窗口 ──────┐
│  Eden (年轻代)    → 新创建的对象     │    │  最新消息      → 最活跃  │
│  Survivor (存活区) → 熬过 GC 的对象  │    │  近期摘要      → 半活跃  │
│  Old (老年代)     → 长期存活对象    │    │  System Prompt → 永久    │
│  GC               → 回收无用对象    │    │  上下文重置    → 释放空间 │
└────────────────────────────────────┘    └─────────────────────────┘
```

### 7.1 映射关系

| JVM 概念 | 上下文工程对应 | 解释 |
|----------|-------------|------|
| 堆最大容量 (-Xmx) | 模型窗口上限 (128K) | 硬顶，超了就 OOM / 截断 |
| 年轻代 (Eden) | 最新一轮消息 | 刚分配，访问频率最高 |
| 老年代 (Old Gen) | System Prompt + 持久摘要 | 长期驻留，不被回收 |
| Minor GC | 滑动窗口截断 | 清理短期无用消息 |
| Full GC | 上下文重置 | 全局压缩，可能「Stop The World」 |
| 内存泄漏 | 无截断的累积历史 | token 只增不减，直到溢出 |
| 对象池 | System Prompt 缓存 | 复用不变部分，减少重复构造 |

### 7.2 一句话

> **用 JVM 调优的思路管上下文——不是垃圾少就不用 GC，是你要设计好「什么进老年代、什么该被回收」。**

## 8. 面试要点

**如果你只有 3 分钟讲上下文工程，讲这 5 句：**

1. **上下文不是越大越好**：128K 窗口塞满反而稀释注意力，信息密度比总量重要
2. **四块分配是动态的**：单轮问答 80% 给文档，Agent 编排 55% 给工具输出，没有一成不变的公式
3. **渐进式披露**：不要把 50 个工具一次性注册，按执行阶段分层注入，类似懒加载
4. **上下文需要 GC**：长对话必须主动截断 / 摘要 / 检查点，否则 token 只增不减、质量退化
5. **文件系统分流**：大批量结果落盘，Prompt 只传路径引用——类比 JVM 栈上放引用、堆上放对象

**一句话总结**（面试金句）：

> 上下文窗口不是给你装满的，是给你设计的。组织方式比窗口大小重要 10 倍。
