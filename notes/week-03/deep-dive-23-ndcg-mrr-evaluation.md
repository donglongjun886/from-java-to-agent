# NDCG/MRR检索评估 — 为什么生成评估不够，需要独立的排序评估

> RAGAS 回答"召回了没"，NDCG/MRR 回答"排序对了没"——这是两个维度的正交评估，缺一不可。

## 关键对比 / 架构认知

NDCG（Normalized Discounted Cumulative Gain）是排序质量的金标准指标。它的核心思想：每个检索位置按对数折扣递减权重，位置越靠后权重越低（1/log2(rank+1)），越靠前的文档对最终得分贡献越大。计算需要多级相关度标注（0-3分），0=无关、1=部分相关（提到但没解答）、2=相关（能回答问题）、3=完美匹配。归一化后 NDCG 落在 [0,1] 区间，便于跨查询横向对比。

MRR（Mean Reciprocal Rank）更简单直接：取首个正确答案出现位置的倒数（1/rank），对所有查询求平均。如果你要找"运费政策"这个问题，第一个相关结果排第3位，本次 MRR 就是 1/3 ≈ 0.33。MRR 天然适合 FAQ 型查询——用户只需要一个正确答案，不关心第二位是什么。

为什么不能只用 RAGAS？两个根本原因：第一，RAGAS 的 Context Recall/Precision 依赖 LLM 做裁判，LLM 自身存在偏好偏差和随机性，同一查询跑两次分数可能不同；而 NDCG/MRR 基于人工标注，具有可复现的客观性。第二，RAGAS 不区分排名差异——召回 Top-5 中，答案排第1位和第5位在 RAGAS 的得分完全一样，但对用户体验是天壤之别。

实际场景选择：探索型查询（"这个产品的竞品分析"）用户会浏览多个结果，用 NDCG 评估排序质量；FAQ 型查询（"退货多少天"）用户只看第一条，用 MRR 评估首位精度。

## Java 映射 + 面试话术

**Java 类比**：NDCG/MRR 相当于后端性能测试中的 P99 延迟——代码 review 和单元测试（类比 RAGAS）能保证代码质量下限，但 P99（类比排序指标）直接决定用户体验上限。RAGAS 是静态代码分析，NDCG 是压测报告——前者告诉你写法对不对，后者告诉你线上表现好不好。

**面试话术**："我们的检索系统评估分两条线：RAGAS 做在线质量监控，NDCG/MRR 做离线排序优化。具体来说，上线前用标注团队标200条查询的3级打分，跑 NDCG@5 和 MRR 基准线。上线后用 RAGAS 持续监控线上数据的上下文召回率和精确率。我印象最深的一次诊断：RAGAS 得分 0.85 看起来不错，但 MRR 只有 0.4——追查发现检索召回的文档虽然包含了答案内容，但排序算法把答案排在第三四位，用户根本看不到。这就是为什么必须有独立的排序评估。"
