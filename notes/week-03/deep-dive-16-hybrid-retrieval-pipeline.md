# 混合检索三级管道 — BM25 → 稠密向量 → Reranker架构设计

> 单一检索算法各有盲区，三级管道是将"召回"和"精排"解耦——这和搜索引擎的L1粗排+L2精排以及广告系统的召回→粗排→精排架构完全同构。

## 关键对比 / 架构认知

**三级管道架构**：第一级是多路召回（Multi-Stage Retrieval）：Query同时走两条路径——BM25文本检索（基于词频-逆文档频率的倒排索引标量匹配，即精确词项匹配）召回Top-50，稠密向量检索（基于语义Embedding的ANN搜索）召回Top-50，两部分结果通过RRF（Reciprocal Rank Fusion）算法融合去重，输出约80-100篇候选文档。第二级是Reranker精排：将融合后的候选集逐一与Query做Cross-Encoder联合编码，计算精确的相关性得分，排序后取Top-5送入LLM。第三级是LLM生成：基于精排后的5篇文档做上下文整合和答案生成。延迟预算分配：BM25检索~10ms，向量检索~20ms（含网络往返），两路并行执行后RRF融合~2ms，Reranker~100ms（将100篇文档打包成一个Batch送入Cross-Encoder做批量推理），总检索延迟约max(10ms, 20ms) + 2ms + 100ms ≈ 122ms——在对话场景的感知阈值（200ms）以内。

**为什么单一检索不够**：BM25基于词项匹配，精确但不理解语义——"心脏病"和"心血管疾病"在BM25眼里毫无关联，但在语义向量空间距离极近。反过来，稠密向量对术语、缩写、代码标识符这类精确匹配很弱——"ErrorCode-500"和"500错误"向量可能相近，但"api_version=v2.3.1"这种版本号字符串，向量模型根本区分不出v2.3.1和v3.0.0的差异。二者互补：BM25保证精确命中不丢，向量保证语义泛化不漏。这是信息检索领域的经典trade-off——Precision（BM25优势）和Recall（向量优势）的平衡。

**为什么还需要Reranker**：这是面试中最容易被追问的点。问题不在于召回不够（100篇通常已经覆盖了正确答案），而在于"100篇中正确答案排在67位"。Reranker使用Cross-Encoder架构，将Query和每篇候选文档拼接后联合编码——这意味着注意力机制可以跨Query和Document计算交互，捕捉细粒度的语义匹配信号（如否定词、条件关系、时间约束）。而Bi-Encoder（向量检索用的双塔模型）将Query和Document分别独立编码，二者的交互仅通过向量点积——丢失了词级、短语级的精确对齐信息。精度差距的本质：Cross-Encoder是"拿着问题去读文档"，Bi-Encoder是"把问题和文档都压缩成向量再比较"。

**工程权衡与优化**：Reranker的100ms是整个管道的瓶颈。三个优化方向：(1) 减少Reranker候选数——BM25和向量检索各自取Top-20而非Top-50，RRF融合后剩30-40篇，延迟显著降低（如降至50-60ms），因GPU推理存在固定开销；(2) 模型蒸馏——用小型Cross-Encoder（如BGE-Reranker-v2-mini，30MB）替代大型模型（BGE-Reranker-v2-large，1.2GB），精度损失<2%，延迟降低60%；(3) 缓存热门Query的Reranker结果——电商/客服场景Query高度集中，Top-100高频查询覆盖50%+流量，缓存命中后Reranker延迟归零。

## Java 映射 + 面试话术

**Java类比**：这套架构在Java生态中有精确对应。BM25 ≈ Elasticsearch的倒排索引+TF-IDF评分，解决精确匹配；稠密向量检索 ≈ 推荐系统中的Embedding Lookup+ANN；RRF融合 ≈ 搜索引擎的Multi-Query多路召回结果合并；Reranker精排 ≈ 广告系统的CTR预估模型（从召回的上千条广告中精排出Top-3），本质都是粗排保召回、精排保准确的两阶段策略。延迟预算分配则完全对应性能调优中的"热点路径分析"——Reranker占77%的延迟（100/130ms），你知道优化重点在哪。这就像用arthas/profile定位到某个Service方法占了整个接口80%的耗时，优化它ROI最高。

**面试话术**："我设计的RAG检索管道是三级架构。第一级多路召回利用BM25和稠密向量的互补性——BM25确保代码片段、版本号、错误码这类精确标识符不被漏掉，向量确保同义表达能被语义覆盖。二者通过RRF融合，这个算法简单但有效：不关心绝对分数，只关心排名，天然解决了BM25和向量分数尺度不一致的问题。第二级Reranker是精排关键——Cross-Encoder的联合编码能力是Bi-Encoder无法替代的，代价是延迟。我的优化策略是缩小融合候选集（各取Top-20）加模型蒸馏，把Reranker延迟从100ms压到30ms以内。第三级LLM接收精排Top-5，在上下文窗口限制下效果最好——控制推理成本、减少上下文噪声避免幻觉、防止超出窗口限制。整套管道的设计思想来自搜索系统的多级排序经验，只是把排序模型从Learning to Rank换成了Cross-Encoder。"
