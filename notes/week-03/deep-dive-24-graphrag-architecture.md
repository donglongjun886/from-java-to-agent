# GraphRAG架构认知 — 知识图谱+RAG的融合模式

> 向量检索解决"像不像"的问题，GraphRAG 解决"是什么关系"的问题——两者互补，不是替代。

## 关键对比 / 架构认知

向量检索的天然短板在关系推理。你问"张三的领导是谁"，向量检索能找到包含"张三"和"领导"这两个词的文档，但它不理解张三和领导之间的「汇报关系」这个结构化事实。多跳推理更难以胜任——"张三个人汇报给李四，李四的直属上级是谁？"这需要两次关系跳转，纯向量检索几乎不可能准确回答。

微软 GraphRAG 方案给出了完整的工程路径：文档 → 实体/关系抽取（LLM 从非结构化文本中提取实体和关系三元组）→ 社区检测（Leiden 算法对图做聚类）→ 层级摘要（对每个社区生成摘要，形成多级知识体系）→ 图遍历查询（用 Cypher 或 Gremlin 做子图检索）。关键在于索引阶段已经把"关系推理"的计算提前做了，查询时只需做轻量级的图遍历。

三种融合模式选型：图谱增强检索（用图谱关系扩展检索条件，比如先查张三的部门节点再检索该部门的文档）适合知识体系清晰、有预定义 Schema 的场景；检索增强图谱（先用向量检索找相关文档，再用 LLM 从这些文档中实时构建小规模子图）适合文档变化频繁、不想维护全局图谱的场景；并行双通道（向量和图谱各自检索然后融合排序）是生产环境最稳妥的方案，两队独立跑、结果按融合策略（RRF 或加权）合并，哪个通道坏了不影响另一个。

代价需要正视：实体/关系抽取的延迟和成本是向量的数十倍、图谱更新需要增量索引策略、Schema 设计需要领域专家介入——不是所有场景都值得上 GraphRAG。

## Java 映射 + 面试话术

**Java 类比**：向量检索 = Elasticsearch 全文倒排索引，查的是文本内容相似度；GraphRAG = 关系数据库的 JOIN + 递归 CTE，查的是实体之间的结构化关系。就像你不会用 ES 替代 MySQL 做多表联查，也不该用向量检索替代知识图谱做多跳推理。

**面试话术**："我们在选型时遵循一个简单判断：用户问的是'像什么'还是'什么关系'。语义相似度问题用向量检索就够了，关系推理密集型查询才上 GraphRAG。去年一个客服项目，用户问'我买的A品类下有哪些关联配件'——这个查询需要遍历商品-配件-兼容性三层关系链，纯向量检索准确率不到40%，引入 Neo4j 存商品关系图谱后准确率提到 85%。但图谱维护成本确实高，所以另一个文档问答项目直接用的向量检索+Chunk 策略，效果也够用。关键是根据查询类型做技术选型，不是无脑上图谱。"
