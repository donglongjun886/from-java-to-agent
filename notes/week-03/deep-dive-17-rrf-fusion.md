# RRF融合算法 — 多路召回结果的标准化融合方案

> RRF的本质不是"排名计算的数学技巧"，而是"在多源异构检索结果之间建立统一比较基准"的融合策略。

## 关键对比 / 架构认知

多路召回（关键词+向量+知识图谱）各自返回带分数的文档列表，但BM25的分数是稀疏的频率加权统计量，余弦相似度是稠密的[-1,1]连续空间度量（主流embedding模型产出的非负向量下，实际范围是[0,1]）——两者分布完全不可比。直接做加权求和，就像把两个月份的收入分别做线性变换和归一化后相加：操作本身成立的，取决于那些月份和基准。因为分数不可直接比，硬加就是让高方差的数据源绑架融合结果。

RRF（Reciprocal Rank Fusion）的做法是忽略分数的数值只保留排名的相对序，核心公式 `score(d) = Σ 1/(k + rank_i)`。其中 k=60 是有讲究的：k 决定了"排名优势的衰减速度"——k 越大，相邻排名差异越小，顶部的极端差距被平滑，更适合召回源质量不稳定、噪声多的场景。排名1和2在具体计算中相差多少？当k=60时，score(1)=1/61，score(2)=1/62，差异仅0.00026；如果换成k=0，差距是1−0.5=0.5，这对排名波动太敏感。60是TREC多年竞赛中表现最稳定的经验值，它意味着"相信排名信号但不过度放大头部优势"。

对比三种替代方案：MinMax归一化对离群点敏感，一个异常高分会让所有其他文档被压缩到0附近；加权求和需要针对每组参数人工调参，且不同检索源的权重在不同查询上并不一致；学习排序（LTR）效果好但需要标注数据且有额外的维护成本。RRF无需调参、无监督、跨任务泛化能力强——这正是工程上追求"简单且鲁棒"的典型设计。

## Java 映射 + 面试话术

**Java 映射**：RRF的融合逻辑可以用"多数据源投票机制"来理解。想象一个电商搜索系统，商品排名由ES关键词搜索、向量召回、协同过滤三个信号合并决定。RRF就相当于给每个数据源的排序结果赋予一个"排名权重票"，再用固定衰减函数（k=60）聚合。这与负载均衡中的加权轮询有精神上的相似：轮询按权重分配请求，RRF按排名分配分数——不同的是轮询的权重是直接配置的，而RRF用排名隐式推导权重。

**面试话术**："我们在多路召回融合时，不直接对BM25分数和相似度分数加和，因为两者量纲和分布完全不同。RRF的做法是忽略分数只看排名，用倒数公式把排名转成统一的可加量。公式里的k=60是一个平滑参数：k太小会让融合结果对单源头部的微小波动过于敏感，k太大会稀释所有排名差异。60这个数来自TREC多年的经验——它让排名1比排名2略占优势但不垄断，也让中尾部排名之间差异被大幅平滑，避免长尾噪声干扰。不管底层是ES、向量库还是图谱，RRF都是一个无监督、零超参的融合层，很适合快速迭代的场景。"
