# 上下文工程 — 渐进式披露 vs 上下文压缩 vs Context Reset

> 上下文工程的本质不是「塞更多信息」，而是管理 LLM 有限且昂贵的注意力预算。

## 关键对比 / 架构认知

长上下文窗口（200K tokens）带来的一个常见误区：既然窗口这么大，把全部文档塞进去不就行了？实际不行——原因有三：（1）注意力稀释：文档越长，LLM 对中间位置的关注度急剧下降（Lost in the Middle 现象），信息塞满反而检索不准；（2）成本飙升：200K 上下文的每次 API 调用成本是 10K 调用的 20 倍，且延迟随之线性增长；（3）幻觉增加：无关信息越多，LLM 越容易被带偏。所以上下文工程的核心问题是：在有限预算内，如何把「最该被注意到的信息」放到 LLM 眼前。

三种策略对应三个场景：

**渐进式披露（Progressive Disclosure）**：先给摘要再给全文，先给结构再给细节。类似前端的懒加载，只有 LLM 明确需要某个文档的细节时才展开。适合长文档问答——先让 LLM 看 5 篇文档摘要，选出最相关的 2 篇再给全文，大幅节省上下文空间。

**上下文压缩（Context Compression）**：用 LLM 或规则压缩历史对话和检索结果。多轮对话中历史轮次越来越长，压缩策略提取每轮的「事实」和「决策」作为摘要，丢弃冗余的中间推理过程。适合长对话场景（客服、Co-Pilot），把 20 轮对话压成 200 token 的摘要段。

**Context Reset**：当对话彻底偏离主题或上下文被错误信息污染，直接重置到干净的初始状态。这比让 LLM 「忽略之前的信息」更可靠——因为 LLM 并不能真正做到选择性遗忘，上下文污染会持续影响后续输出。适合话题切换或错误纠正后重新开始。

## Java 映射 + 面试话术

**Java 类比**：渐进式披露 = Hibernate Lazy Loading——先加载代理对象（摘要），真正用到属性时才查库（给全文）；上下文压缩 = JVM GC 的内存整理——压缩活跃对象（关键事实），回收垃圾（冗余推理过程）；Context Reset = 数据库连接池的废弃连接重建——连接被污染了不修，直接扔掉换新的。

**面试话术**："长上下文不能替代 RAG 的根本原因在于注意力预算管理。我们的组合策略：检索阶段用 RAG 从 10 万篇文档召回 Top-10 候选（粗筛），上下文阶段用渐进式披露——先给 LLM 看每篇的标题和摘要（约 500 token），LLM 确认需要哪 3 篇的全文后再展开（约 3000 token），总消耗控制在 4000 token 以内。多轮对话时加入压缩模块，每 5 轮自动摘要压缩一次历史，窗口利用率保持在 60% 以上。这套组合拳比一股脑塞 200K 上下文效果更好，成本还低一个数量级。"
