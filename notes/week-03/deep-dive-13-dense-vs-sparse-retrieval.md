# 稠密检索 vs 稀疏检索：互补而非替代的工程论证

> 给检索场景一律上向量化是一厢情愿——稀疏检索管精确匹配，稠密检索管语义泛化，二者各有盲区，混合才是工业界的工程甜点。

## 关键对比 / 架构认知

稀疏检索（BM25/Lucene）的核心优势在三个场景：**精确匹配**——订单号、身份证号、代码变量名，倒排索引毫秒级命中；**专有名词**——"Spring Cloud Gateway"，分词后精确锁定，Embedding模型可能未见过这个词组合；**零样本检索**——不需要训练数据，即开即用。代价是**词汇不匹配问题**：用户搜"笔记本散热差"，文档写"电脑温度高"，倒排索引认为完全无交集。

稠密检索（Embedding + 向量搜索）恰恰相反。优势在**语义泛化**——"性价比高的手机"和"便宜好用的智能机"余弦相似度接近1；**跨语言检索**——英文query检索中文文档；**口语化容错**——错别字、口语表达自动对齐。但盲区同样突出：**冷启动成本**——需要大量标注数据训练或微调Embedding模型；**领域迁移（domain shift）**——通用Embedding模型在垂直领域术语上的语义空间是扭曲的，效果断崖下跌；**不可解释**——召回了但说不清为什么，排查Bad Case像黑盒。

两者的工程属性形成完美互补：BM25保底——任何场景都能给出不差的基线召回；Embedding补充——在语义模糊查询上补全BM25的盲区。融合策略上，**RRF（Reciprocal Rank Fusion）**是最简洁有效的方案：对两个检索链路的Top-K做加权融合，公式为 `score(d) = Σ 1/(k + rank_i(d))`，k通常取60，无需任何归一化即可融合稀疏和稠密的异构分数。

## Java 映射 + 面试话术

映射：这就像MySQL的全文索引（MATCH AGAINST做精确词匹配）和向量数据库（Milvus/Qdrant做语义搜索）的互补。MySQL命中精确关键词快且准，但用户搜"数据库连接池原理"时，含"HikariCP配置优化"的文档MySQL全文索引找不到——向量检索负责这个语义缺口。RRF融合层的角色类似搜索引擎的LTR（Learning to Rank）排序层，但更轻量。

**面试话术**："我的判断是混合检索是唯一工程选择。BM25在专有名词、精确ID、零冷启动场景是碾压级的——这些场景向量检索要么召回不全，要么需要昂贵的领域微调。向量检索在口语化查询和语义泛化上是BM25补不了的盲区。我的工程策略是：BM25做第一路召回保证底线，Embedding做第二路语义扩充，RRF融合两路Top-K，不需要做分数归一化。从成本角度，绝大多数通用检索场景BM25已经给出可用基线，向量检索是那5%模糊查询的增值保险。架构上就是MySQL全文索引加向量数据库双引擎，跟Java后端的多级缓存思路一个道理。"
