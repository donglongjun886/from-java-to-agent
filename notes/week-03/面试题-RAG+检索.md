# 面试题：RAG + 信息检索

> 对标 Zoom Enterprise RAG 资深研发工程师 JD，面向 12 年 Java 后端转型 AI Agent 的面试准备。

## 板块一：RAG 架构与工程实践

### Q1: RAG 的核心链路是什么？每个环节有哪些技术选型？

**考察点**: 全链路架构

**参考答案**: RAG = 先找后读：检索找依据，生成写答案。六阶段：①文档摄入（Unstructured/Tika/LangChain Loader）；②文档切分（递归切分/语义切分，512-1024 token）；③向量化（BGE-M3/text-embedding-3）；④索引存储（PGVector 起步/Milvus 规模化）；⑤检索（Top-K→混合检索→Reranker 精排）；⑥生成增强（注入 Prompt + LLM 生成）。工程化还需评估环（RAGAS）和可观测层（Langfuse）。

### Q2: 混合检索（BM25 + 稠密向量）为什么比纯向量检索好？什么场景下纯向量就够了？

**考察点**: 检索策略权衡

**参考答案**: 混合检索解决的是单一检索范式的盲区问题。稠密向量检索擅长语义匹配，比如「怎么退钱」能匹配到「退款流程」文档，但它对精确关键词匹配不敏感，搜「JVM-1024」可能召回 JVM 通用文档而非具体错误码页面。BM25 稀疏检索正好反过来——关键词匹配精准，但无法理解同义词和语义变体。混合检索取两者并集后按 RRF 融合排序。纯向量检索在两种场景下足够：一是文档语料本身就是语义密集型（如 FAQ、产品介绍），关键词歧义少；二是使用高质量 Embedding 模型（如 BGE-M3）且领域数据充分微调过，已在目标数据上验证了召回率达标。

**Java 类比**: 像 Elasticsearch 的 BM25 全文检索 + dense_vector 字段的 kNN 搜索混合打分——一个做精确匹配，一个做语义相似。

### Q3: Reranker 在 RAG pipeline 中起什么作用？通常放在哪个位置？

**考察点**: 检索质量优化

**参考答案**: Reranker 是检索后的精排模块，对初召回的候选文档（通常 Top-50 到 Top-100）做二次打分，输出最终送入 LLM 的结果（Top-3 到 Top-5）。它解决初召回「排不准」的问题：BM25 可能把关键词命中多的长文档排前面但无关，Embedding 可能把语义近似但不含答案的文档排前面。Reranker 用 Cross-Encoder 架构，把 Query 和每个候选文档配对输入模型做深度相关性判断，精度远高于 Bi-Encoder（Embedding 模型）。主流选择是 Cohere Rerank、BGE-Reranker 或 Jina Reranker。位置必须在检索之后、生成之前，延迟开销可控（通常 50-200ms）。从架构角度，Reranker 是典型的「粗排→精排」两阶段策略，和搜索引擎广告系统如出一辙。

### Q4: 企业级 RAG 的多租户隔离怎么做？索引级 vs 元数据过滤级各自的优劣？

**考察点**: 多租户架构

**参考答案**: 两种方案。索引级隔离（Physical Partition）为每个租户建独立向量集合/索引，优点是安全边界硬隔离，故障不扩散，性能互不干扰；代价是租户数多时元数据管理复杂，共享文档无法去重存储。元数据过滤（Logical Filter）所有租户数据存同一索引，检索时 WHERE tenant_id = X 过滤，优点是运维成本低，跨租户知识可共享；缺点是过滤在检索后执行时可能漏召回（向量检索结果中目标租户数据排不进 Top-K），必须在查询时就带过滤条件（pre-filtering）。实战中建议：租户数 < 100 用索引隔离，租户数多且数据量小的 SaaS 场景用元数据过滤 + pre-filtering。最稳妥的是 Milvus 的 Partition Key 特性——物理上分片、逻辑上同一集合。

**Java 类比**: 索引级隔离 = 分库，元数据过滤 = 单库 + tenant_id 行级过滤。

### Q5: 文档切分（chunking）策略有哪些？如何选择 chunk size 和 overlap？

**考察点**: 数据预处理

**参考答案**: 切分策略分固定规则和语义感知两类。固定规则包括字符切分（按 token 数截断）、递归字符切分（按自然分隔符优先级 "\n\n" > "\n" > "。" 逐级回退）、句子切分。语义感知包括基于 Embedding 相似度的语义边界检测、按 Markdown 标题层级切分、按文档原始结构切分。参数选择：chunk size 太小（<256 token）导致上下文碎片化，LLM 看不到完整逻辑；太大（>1024 token）稀释检索精度，且影响生成质量（无关内容占上下文窗口）。实践中 512-1024 token 是通用舒适区。overlap 设置 10%-20%，保证跨 chunk 边界的实体/概念不丢失。关键原则：chunk 应当是一个「自包含的语义单元」，一个函数定义、一个 API 文档段落就是一个好 chunk。高级方案还有 LlamaIndex 的 small-to-big retrieval：用小 chunk 做精确检索，但返回大 chunk 或完整段落给 LLM，兼顾检索精度和上下文完整性。


### Q6: RAG 的检索权限控制为什么必须在检索阶段做，不能等 LLM 生成后再过滤？

**考察点**: 安全架构

**参考答案**: 两个致命问题。一是信息泄露风险：如果在 LLM 生成后再过滤，敏感文档内容已经进入 LLM 的上下文窗口，模型可能在不相关回答中「不经意间」引用这些信息。这等同于数据泄露事故，在 SOC2/ISO27001 审计中一票否决。二是上下文窗口浪费：无权限文档挤占了有限的上下文窗口预算，导致有效检索结果被挤出，回答质量下降。正确的做法是 pre-retrieval filtering——在查询阶段就把权限条件作为检索过滤器传入，保证检索结果集中只包含当前用户有权访问的文档。实现上，可以在元数据中存储 ACL（Access Control List）字段，检索时用 $in 过滤或构建权限向量匹配。

### Q7: 如何处理异构数据源（PDF + 数据库 + API）的统一摄入？

**考察点**: 数据管道设计

**参考答案**: 核心思路是抽象统一的数据模型（Unified Document Model），将不同数据源映射为统一的 Document Schema：doc_id、source_type、content、metadata、embedding、permissions。PDF 用 Unstructured 或 PyMuPDF 解析，需处理表格、图片、双栏布局等复杂场景。数据库通过 CDC（Change Data Capture）增量同步，Debezium 监听 binlog → Kafka → 消费端重新向量化。API 数据源用定时轮询或 Webhook 触发，状态变更后触发重新索引。管道设计上推荐提取-转换-加载（ETL）触发层 + 向量化 Worker 的解耦架构，类似 Java 中消息驱动 + 异步处理的模式。关键工程挑战不在单个解析器，而在统一的质量保障（解析失败重试、脏数据隔离）和增量更新（已删除文档的向量索引清理）。

**Java 类比**: Canal + Kafka + 异步刷 Cache 的标准数据同步模式。

### Q8: RAG 的幻觉问题有哪些缓解手段？

**考察点**: 质量保障

**参考答案**: 三类手段叠加使用效果最好。第一是引用溯源：强制 LLM 在回答中标注信息来源 chunk_id，前端渲染时展示原文引用链接。实现上在 Prompt 中要求输出 JSON 格式引用，用结构化输出约束。第二是事实核查：让 LLM 生成回答后，用另一个 LLM 调用来检查生成内容的每个断言是否在检索文档中有支撑。LangChain 的 FactualEval 和 LlamaIndex 的 FaithfulnessEvaluator 都提供实现。第三是拒答策略：当检索结果相关性评分低于阈值（如所有文档 score < 0.5）时，直接返回「知识库未收录相关信息」，拒绝生成。关键工程决策是拒答阈值：设太严格频繁拒答体验差，设太宽松容易幻觉。建议用 RAGAS 评估体系离线标定阈值，上线后持续监控调整。

### Q9: Langfuse 在 RAG 系统中扮演什么角色？和 RAGAS 怎么分工？

**考察点**: 可观测性

**参考答案**: Langfuse 是 LLM 应用的可观测平台，RAGAS 是 RAG 系统的评估框架，职责不同。Langfuse 做线上监控：Trace 粒度记录每一次 RAG 调用链（检索延迟、Token 消耗、LLM 耗时、生成结果），类似 Java 中 SkyWalking + ELK 的角色。RAGAS 做离线评估：基于标注数据集，计算 Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision（上下文精度）等指标。分工上，RAGAS 回答「系统效果好不好」，Langfuse 回答「系统哪里慢了/出错了」。实战中两者组合：用 RAGAS 在发版前做离线评估建立基线，用 Langfuse 在生产环境实时监控异常和劣化。Langfuse 还支持用户反馈收集和 A/B 实验，这对持续优化检索策略至关重要。

### Q10: 上下文窗口越来越大（200K+），RAG 还有必要吗？什么场景下仍然需要？

**考察点**: 架构判断力

**参考答案**: 仍然需要，三个场景无法被长上下文替代。一是海量文档场景：企业知识库动辄上亿 Token，远超任何上下文窗口上限。即使长窗口模型能塞入整本书，成本也是灾难——每次请求都携带全量文档的 Token 消耗不可接受。二是精准度需求：大窗口不等于大窗口内搜索能力强。研究表明，LLM 在多文档信息定位上的精度随文档数量和位置变化显著衰减（Lost in the Middle 效应），检索系统能精确找到相关段落。三是延迟与成本：200K Token 的推理延迟远高于 10K Token + 检索的延迟，API 成本差距在 5-10 倍。结论是长上下文是 RAG 的补充而非替代——用 RAG 筛选相关文档，用长上下文承载筛选后的大篇幅参考材料，两者组合才是最优方案。

## 板块二：信息检索基础

### Q11: 倒排索引的原理是什么？为什么它比全表扫描快？

**考察点**: 检索数据结构

**参考答案**: 倒排索引（Inverted Index）是搜索引擎的核心数据结构。正向索引是文档→词的映射（这篇文档包含哪些词），倒排索引反过来：词→文档列表的映射（这个词出现在哪些文档）。查询时按查询词直接定位到文档列表，只处理命中文档，不需要遍历全部文档。性能优势来自三个层面：一是跳表或 BitMap 结构做交集运算，词典查找近乎 O(1)，只需处理命中文档的倒排列表，避免遍历全部文档；二是词典常驻内存，磁盘只读取命中的倒排列表；三是 Skip List 和 Block-Max 索引可以跳过大量非相关文档。Elasticsearch 底层 Lucene 就是倒排索引的经典实现。在 Java 中 Hashmap O(1) 查找 + 倒排列表求交就是 BM25 检索的核心计算路径。

**Java 类比**: HashMap<Term, List<DocId>> 实现 O(1) 词典查找。

### Q12: TF-IDF 和 BM25 的核心差异是什么？BM25 做了哪些改进？

**考察点**: 经典算法

**参考答案**: TF-IDF 有两个核心缺陷，BM25 全部修复。一是 TF 饱和：TF-IDF 中词频线性增长，一篇提到 100 次「Java」的文档得分是提到 1 次的 100 倍，这过度放大了重要性。BM25 用非线性饱和函数控制词频贡献——词频超过一定次数后边际增益递减（TF/(k1+TF) 形态），而非 TF-IDF 的线性增长。二是文档长度惩罚：TF-IDF 没有长度归一化，长文档天然词频高而得分虚高。BM25 用 b 参数控制长度归一化程度，b=0 无视长度，b=1 完全归一化，默认 0.75。另外 BM25 的 IDF 做了平滑修正（+1），防止文档频率过半时出现负权重，高频词不会被过度惩罚。效果上 BM25 在几乎所有 TREC 基准上显著优于 TF-IDF，是稀疏检索的事实标准。

### Q13: 稠密检索（Embedding）和稀疏检索（BM25）各自的优劣势？

**考察点**: 检索范式对比

**参考答案**: 稠密检索优势是语义泛化——「高效液相色谱」和「HPLC」在 Embedding 空间天然相近，不需要同义词词典。对跨语言检索、口语化查询容错强。劣势是训练/推理成本高，模型领域适配需要 fine-tune，冷启动差（新来的实体 Embedding 无法匹配）。稀疏检索优势是精确匹配、可解释性强（能明确说因为命中哪些词而召回）、零样本就能用（无需训练），对专有名词、错误码、ID 类查询效果远超稠密检索。劣势是词汇不匹配问题（查「跑路」匹配不到「卷款潜逃」）。所以混合检索是当前主流：BM25 保底精确召回，Embedding 补充语义召回，RRF 融合并集排序。工程上多路召回 + 融合排序是性价比最高的架构。

### Q14: NDCG 和 MRR 分别衡量检索质量的什么维度？

**考察点**: 评估指标

**参考答案**: NDCG（Normalized Discounted Cumulative Gain）衡量排序质量，MRR（Mean Reciprocal Rank）衡量首个相关文档的位置。NDCG 关注整体排名是否合理——相关文档是否排在前面，每个位置的贡献按对数折扣，越靠后权重越低。它需要标注相关度等级（0-3 分），而不只是二分类相关/不相关，是对排序质量的综合打分。MRR 只看第一个相关文档出现的位置，取倒数后求平均。MRR 高说明用户不需要翻很久就能看到答案，适合 FAQ 搜索这类「有唯一正确答案」的场景。NDCG 适合知识库探索类场景，用户会看多个结果。两者互补：NDCG 反映整体排序质量，MRR 反映用户获取答案的效率。

### Q15: RRF（Reciprocal Rank Fusion）是怎么融合多个检索结果的？

**考察点**: 融合算法

**参考答案**: RRF 是多路检索结果的无监督融合算法。公式简单：每个文档的 RRF 分数 = 对所有检索源求和 1/(k + rank)，其中 k 是常数（通常取 60），rank 是该文档在该检索源中的排名。核心思想是不依赖分数的绝对大小（因为不同检索源的分数分布差异大，BM25 分数和余弦相似度无法直接比较），而是只依赖相对排名作融合。k=60 的设计理念是：排名前 10 的文档权重差异大（1/61 ≈ 0.016 到 1/70 ≈ 0.014），排名靠后的文档权重趋于平坦但仍有区分度。相比简单的大平均或 MinMax 归一化，RRF 不需要训练、不引入超参调优，在 TREC 竞赛中表现稳定，是混合检索的标准融合方案。

### Q16: Embedding 模型的选型依据是什么？

**考察点**: 模型评估

**参考答案**: 四个维度。一是 MTEB 基准得分：看 Retrieval 任务子榜而非总分，因为分类和聚类表现好不等于检索好。注意 MTEB 数据集以英文为主，中文场景需参照 C-MTEB。二是维度和性能：同等训练条件下，更高维度通常带来更强的表达能力，但存储和检索成本线性增长，且训练质量比维度更关键。生产中常用 Matryoshka 表示学习技术，使模型支持可变维度的同时保证精度。三是领域适配：通用模型在医疗、法律、金融等专业领域表现下降明显，需要评估是否需微调。BGE-M3 和 GTE 系列在中文场景表现好，E5 系列在英文场景通用性强。四是推理延迟：高吞吐场景下 QPS 是关键指标，需选择适合生产部署的模型。建议：从 BGE-M3 或 text-embedding-3-large 起步，在自有测试集上横向对比。

### Q17: 向量数据库（ChromaDB / Milvus / PGVector）选型的核心考量是什么？

**考察点**: 基础设施

**参考答案**: 三个维度。一是规模与性能：数据量 < 10 万且以原型为主选 ChromaDB（轻量、Python 生态友好）。数据量 < 100 万条选 PGVector（复用 PostgreSQL，运维成本最低）；10 万 - 100 万条选 Qdrant 或 Weaviate（专用向量库，HNSW 索引性能好）；亿级以上选 Milvus（分布式架构，支持 GPU 加速索引构建）。二是功能需求：是否需要标量过滤 + 向量检索的组合查询（filtered search），Milvus 和 Qdrant 支持更好；是否需要多模态向量（图文混合检索），是加分项。三是运维复杂度：PGVector 最大优势是 DBA 团队现有能力可覆盖，不用引入新基础设施；Milvus 需单独部署维护，但性能天花板高。Zoom 级别的企业 RAG 建议 PGVector 起步 + Milvus 预留——数据量突破百万后平滑迁移，避免过早优化。

### Q18: Agentic Retrieval 和静态 RAG Pipeline 的本质区别是什么？

**考察点**: 架构演进

**参考答案**: 静态 RAG 是固定的 DAG（有向无环图）：查询 → 检索 → 重排 → 生成，每一步的输出是下一步的输入，路径固定。Agentic Retrieval 引入 LLM 作为推理引擎做动态路由：LLM 根据查询意图自主决定调用哪个检索工具（向量库、知识图谱、计算器、外部 API）、调用几次、以什么策略融合结果。本质区别在于控制流从静态编排变为 LLM 驱动的动态决策。适合 Agentic 的场景：多步推理问题（先查 A 再根据 A 的结果查 B）、工具选择问题（查文档 vs 查数据库 vs 调用 API）、自适应检索（检索结果不够时自动改写查询重试）。不适合的场景：Q&A 型单一检索，固定 Pipeline 更稳定可控。

**Java 类比**: 静态 RAG = 固定工作流（Flowable 硬编码 BPMN）；Agentic RAG = LLM 作为编排器自主决定调用链，类似 AI 驱动的工作流引擎。

### Q19: 检索评估和生成评估为什么要分离？只用 RAGAS 有什么盲区？

**考察点**: 评估体系

**参考答案**: 检索和生成是两个不同层次的问题，混淆评估会导致无法定位根因。举例：最终回答错误，可能是检索本身就没召回相关文档（检索问题），也可能是检索对了但 LLM 理解偏了（生成问题）。分离评估意味着对检索输出和生成输出分别打分：检索侧用 NDCG、MRR、Recall@K 评估召回质量和排序质量；生成侧用 Faithfulness（忠实度）和 Answer Relevancy 评估回答质量。RAGAS 的盲区在于：它的 Context Recall 和 Context Precision 依赖 LLM 做裁判打分，不是真实的人工标注，存在评估者偏差（evaluator bias）。而且 RAGAS 不能回答「用户满意度如何」「延迟是否可接受」等在线指标。结论：RAGAS 做离线快速评估，但必须辅以人工标注测试集 + 在线 A/B 指标形成完整评估环。

### Q20: 知识图谱（GraphRAG）解决什么问题？和向量检索是互补还是替代？

**考察点**: 前沿方案

**参考答案**: GraphRAG 解决向量检索在关系推理上的天然短板。向量检索擅长「语义相似」——问 AI 伦理能找到 AI 伦理的文档。但它无法理解结构关系——「CEO 张三的直接下属有哪些人」「某药物和某疾病的传递路径」这类多跳推理问题，向量检索几乎没有召回能力。GraphRAG 将文档抽取为实体和关系构建知识图谱，查询时用图遍历算法沿关系边做多步推理。它是互补关系而非替代：向量检索负责找到相关文档，知识图谱负责从文档间提炼结构化关系做推理。微软 GraphRAG 方案的典型架构：先向量检索召回候选文档集，再对文档集做实体提取构建局部图谱，最后在图谱上做社区检测和摘要生成。工程上引入知识图谱的代价是数据抽取和更新的复杂度显著增加，适合关系密集型场景（企业组织架构、政策法规、科研文献）。

**Java 类比**: 向量检索是全文索引，GraphRAG 是外键 JOIN + 递归 CTE——解决的是结构关系查询问题。
