# 代码沙箱选型：E2B vs Docker vs Wasm

> 沙箱选型的核心权衡不是"安不安全"，而是"安全到什么程度"和"启动有多快"——Docker 够用 80% 场景，E2B 是 Agent 场景的最佳体验，Wasm 是未来的轻量方案。

## 关键对比

| 维度 | Docker 容器 | E2B（Firecracker microVM） | Wasm（WebAssembly） |
|------|------------|---------------------------|---------------------|
| 隔离级别 | 容器级（共享内核） | VM 级（独立内核，硬件虚拟化） | 进程级（Wasm 运行时沙箱） |
| 启动速度 | 1-3 秒 | ~150ms（microVM 启动，不含环境初始化） | <1ms（无冷启动） |
| 安全强度 | 中（内核共享有逃逸风险） | 高（真隔离，Firecracker 设计目标） | 中高（语言级沙箱，无系统调用访问） |
| 文件系统 | 持久（volume 挂载） | 临时（microVM 销毁即清空） | 无传统 FS（线性内存） |
| LLM 生成代码 | 任意语言 | 任意语言 | 多语言但受限于 Wasm 运行时 API（Python 可用 Pyodide，但无法执行 `os.system()` 等系统调用） |
| 成本 | 低（Docker daemon 常驻） | 中（按 microVM 使用时长） | 极低（嵌入宿主进程） |
| 适用场景 | CI/CD、本地开发 | Agent 即时代码执行 | 边缘计算、浏览器沙箱 |

**为什么 Agent 需要专门的沙箱**：
Agent 执行 LLM 生成的代码有独特要求：
- **任意语言**：LLM 可能生成 Python/Bash/JS，不能限制语言
- **瞬时生命周期**：执行完即销毁，不需要持久化
- **极速启动**：用户等 3 秒启动 Docker 容器体验很差
- **网络控制**：允许下载包但不能访问内网

E2B 专为这个场景设计：Firecracker microVM 150ms 启动，执行完自动销毁，支持任意语言，自带网络隔离。

## Java 映射 + 面试话术

**Java 类比**：
- Docker 容器 ≈ **独立 JVM 进程（共享宿主机内核）**——namespace + cgroups 隔离，比 ClassLoader 可靠得多，但内核共享仍有逃逸风险
- E2B microVM ≈ **独立 JVM 进程**——完全隔离，自己的堆、自己的 GC
- Wasm ≈ **JVM 字节码沙箱（概念上类似 SecurityManager，JDK 17 已弃用）**——语言层面的受限执行环境，不能访问系统资源

**面试这么说**：
> "代码沙箱选型我分四级看。最轻的是 Wasm，像 JVM 的字节码沙箱——语言级安全，启动极快，但限制了语言选择。日常开发用 Docker 容器足够了，跟 CI/CD 的构建隔离一样。但如果 Agent 执行的代码是 LLM 生成的、不确定来源的，我会用 E2B——Firecracker microVM 真隔离，150ms 冷启动，执行完即销毁，不会污染宿主机。最严格的是物理机隔离（air-gapped），只在金融/军工场景有用。绝大多数 Agent 场景 E2B 就够了，性价比和安全性的甜点级选择。"
