# Agent 安全三层防线：Prompt注入防御 → Tool预检 → 输出校验

> Agent 安全不能用"一道墙"解决——Prompt 注入发生在输入端，危险 Tool 调用发生在执行层，敏感信息泄露发生在输出端。需要纵深防御。

## 关键认知：为什么传统 API 安全模型不够用

传统 Web 安全（OWASP Top 10）假设"输入是用户的数据，代码是开发者写的"。Agent 颠覆了这个假设——**LLM 的输出变成了新的攻击面**。如果 LLM 被注入 Prompt 诱导调用危险 Tool，传统 WAF 基于规则匹配，无法理解 Prompt 的语义攻击意图，对越狱和角色劫持基本无效。

## 三层防线架构

```
用户输入 ──→ [第1层: Prompt 防火墙] ──→ LLM推理 ──→ [第2层: Tool 预检] ──→ 执行 ──→ [第3层: 输出校验] ──→ 响应用户
                ├─ 注入检测                   ├─ 参数白名单              ├─ PII/密钥脱敏
                ├─ 角色越狱检测               ├─ 频率限制                ├─ 引用溯源校验
                └─ 敏感词过滤                 └─ 危险操作二次确认        └─ 幻觉事实核查
```

| 层级 | 防御什么 | 失效后果 | 典型方案 |
|------|---------|---------|---------|
| Prompt 防火墙 | 越狱攻击、角色劫持、间接注入 | 后续防线全部旁路 | 独立 LLM 做输入分类（AEGIS/GuardRails） |
| Tool 预检 | 恶意参数、越权调用、高频滥用 | 数据泄露/资源耗尽 | 参数白名单 + Rate Limiter + 权限校验 |
| 输出校验 | 幻觉、敏感信息泄露、注入下游 | 用户被误导/供应链攻击 | 事实核查 Agent + PII 正则 + 引用对齐 |

## Java 映射 + 面试话术

**Java 类比**：
- 三层防线 ≈ **Spring Security Filter Chain**——每个 Filter 独立职责，链式处理，任一层拦截即短路
- Tool 预检 ≈ **API Gateway 的请求校验**——参数白名单、限流、权限鉴权
- 输出校验 ≈ **Spring MVC 的 `ResponseBodyAdvice`**——在响应写出前对 body 做统一拦截处理，不管内部怎么算，出去之前必须过一遍校验规则

**面试这么说**：
> "Agent 安全我用纵深防御模型，跟 Spring Security Filter Chain 一样三层。第一层 Prompt 防火墙，用独立小模型做输入意图分类——这层被绕过，后面全白费。第二层 Tool 预检，跟 API Gateway 的请求校验一样，参数白名单 + 权限 + 限流，特别是危险操作（发邮件、删除数据）必须二次确认。第三层输出校验，防幻觉和敏感信息泄露。为什么不能只用一层？因为攻击面不同——Prompt 注入在输入侧，LLM 幻觉在生成侧，工具滥用在执行侧，一层墙挡不住三个方向的攻击。"
